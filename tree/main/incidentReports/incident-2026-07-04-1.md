# Incident: 2026-04-07 14:08:00

## Summary

Between 14:08 and 14:19 on 2026-04-07, all users attempting to order pizzas through the JWT Pizza service encountered order fulfillment failures. The event was triggered by a chaos injection at 14:08 UTC by the JWT Pizza Factory API. The factory began rejecting all incoming order requests, returning error responses instead of fulfilled orders.

A hard dependency on the external Pizza Factory API caused 100% of pizza orders to fail with the message `"Failed to fulfill order at factory"`. The event was detected by a Grafana alert rule monitoring `pizza_failed_ratio`. The incident was resolved by following the `followLinkToEndChaos` URL embedded in the factory error response. This high severity incident affected all pizza ordering functionality for approximately 11 minutes.

## Detection

This incident was detected when the **"Pizzas failed"** Grafana alert rule was triggered and the on-call engineer was notified via email.

The alert rule monitored:
```promql
increase(pizza_failed_ratio{source="jwt-pizza-service"}[5m])
```

The alert fired within 1 minute of the chaos starting, at approximately 14:09. Detection time could be improved by also setting up SMS or push notification alerts in addition to email, to ensure faster acknowledgement.

## Impact

For approximately 11 minutes between 14:08 UTC and 14:19 UTC on 2026-04-07, the JWT Pizza ordering system was completely unavailable. All pizza order fulfillment requests failed during this window.

This incident affected all users attempting to place pizza orders during the incident window. No support tickets were submitted as the system was running simulated traffic during this period. Revenue loss was limited to simulated orders only.

## Timeline

All times are UTC.

- _14:08_ - Error logs begin showing `error` level factory log entries with the chaos error message.
- _14:09_ - Grafana "Pizzas failed" alert fires. On-call engineer receives email notification.

## Response

After receiving an email alert at approximately 14:09 UTC, the on-call engineer came online immediately and began investigating via the Grafana dashboard. The engineer identified the spike in `pizza_failed_ratio` and cross-referenced the logs to find the factory error message containing the resolution link. The incident was resolved within 11 minutes of first impact.

Response time could be improved by configuring SMS or mobile push notifications in addition to email, ensuring faster acknowledgement if the engineer is away from their computer.

## Root cause

The external JWT Pizza Factory API began rejecting all order fulfillment requests as part of an intentional chaos injection. The JWT Pizza Service has a hard dependency on the Pizza Factory API with no fallback mechanism — when the factory rejects requests, 100% of pizza orders fail with no graceful degradation or retry logic.

## Resolution

The factory error response contained a `followLinkToEndChaos` field:
```
https://pizza-factory.cs329.click/api/support/3d6ec12e2da9491a8150ee0cbd28bb64/report/i9tqi8bw
```

Navigating to this URL signaled the Pizza Factory to end the chaos injection and resume normal order fulfillment. No application code changes were required. Normal operation was confirmed by observing `pizza_failed_ratio` returning to 0 in Grafana and successful factory log entries reappearing.

## Prevention

This was the first chaos injection incident for this system. The error message format — containing a `followLinkToEndChaos` field — is a known pattern from the JWT Pizza Factory and should be documented in a runbook so future engineers can resolve similar incidents faster.

## Action items

1. Create a runbook documenting the `followLinkToEndChaos` resolution process for factory chaos events, so any on-call engineer can resolve it without needing to diagnose from scratch.
2. Add SMS/push notification to the Grafana OnCall contact point to improve time-to-acknowledgement.
3. Investigate adding retry logic or a graceful degradation response (e.g. queuing orders) when the Pizza Factory API returns errors, to reduce user impact during future factory outages.
